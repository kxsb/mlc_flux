|1. CONNEXION ET EXTRACTION CYCLOS| | | | |
|Configuration de l’instance Cyclos|server/config.py|CYCLOS_BASE_URL, CYCLOS_USERNAME, CYCLOS_PASSWORD ; chargés depuis la config / .env|C’est ici qu’on indique à MLCFlux quelle API Cyclos interroger et avec quels accès techniques. Ce fichier ne récupère pas les données lui-même : il donne les paramètres de connexion.|« La base URL et les identifiants sont centralisés dans config.py ; le client API les réutilise ensuite. »|
|Construction de l’authentification|server/services/cyclos_client.py|build_basic_auth(...)|Le client prépare l’authentification nécessaire pour demander une session à Cyclos.|« On ne met pas la logique d’accès dans les vues : elle est isolée dans le client Cyclos. »|
|Ouverture de session Cyclos|server/services/cyclos_client.py|create_session_token(...)|MLCFlux ouvre une session auprès de Cyclos et récupère le jeton utilisé pour les requêtes suivantes.|« La récupération passe d’abord par une session Cyclos dédiée, puis les appels utilisent le token obtenu. »|
|Endpoint Cyclos utilisé pour les transactions|server/services/cyclos_client.py|get_transactions(...) ; appel GET /transactions|La récupération principale des flux monétaires part de l’endpoint transactions de Cyclos.|« Pour les flux, la source primaire est /transactions. »|
|Période interrogée côté Cyclos|server/services/cyclos_client.py|get_transactions(...), paramètre datePeriod=[start, end]|MLCFlux demande explicitement une fenêtre de dates à Cyclos. Cela évite de charger tout l’historique à chaque appel.|« On interroge Cyclos par période, jamais toute l’historique à l’aveugle. »|
|Dates saisies en jours civils français|server/services/cyclos_client.py|_is_date_only(...), _parse_date(...), logique Europe/Paris|Quand on fournit 2026-05-15, MLCFlux comprend « le jour local français », puis convertit correctement pour l’API. Cela évite les décalages lors des changements d’heure.|« Les bornes de dates sont normalisées depuis Europe/Paris avant d’être envoyées à Cyclos. »|
|Date de fin inclusive côté interface|server/services/cyclos_client.py|get_transactions(...), traitement de date_to|Si on demande jusqu’au 15 mai inclus, le client transforme proprement la borne avant envoi à l’API, afin de ne pas perdre les transactions du dernier jour.|« date_to est traité comme inclusif pour l’utilisateur. »|
|Pagination complète des transactions|server/services/cyclos_client.py|get_transactions(...), page, pageSize=500, X-Has-Next-Page|Cyclos renvoie les résultats par pages. MLCFlux boucle jusqu’à avoir toutes les pages de la période, au lieu de ne prendre que les 500 premières transactions.|« La pagination est explicite ; le bug initial des 500 premières lignes a été corrigé. »|
|Pourquoi l’interface ne requête pas Cyclos en direct|server/sync_transactions.py ; server/analytics.py|sync → SQLite → analytics|Les analyses lourdes s’appuient sur une base locale anonymisée et synchronisée. Cela évite les requêtes API massives à chaque chargement de page et réduit le risque de timeout.|« L’API sert à synchroniser ; l’interface lit ensuite une base analytique locale. »|
|Lancement manuel d’une synchronisation|server/sync_transactions.py|CLI : --days, --date-from, --date-to|On peut synchroniser une fenêtre courte ou relancer une période précise pour corriger ou compléter l’historique.|« La sync est paramétrable par fenêtre temporelle. »|
|Backfill historique complet|script d’exploitation autour de server/sync_transactions.py|Ex. backfill année par année sur 2019-04-17 → aujourd’hui|Pour reconstituer l’historique, on lance des synchronisations découpées par périodes, plutôt qu’un unique appel gigantesque.|« Le backfill complet est un enchaînement de syncs datées, ce qui est plus robuste. »|
|2. CE QUI EST STOCKÉ ET COMMENT C’EST MIS À JOUR| | | | |
|Champs transactionnels conservés en SQLite|server/sync_transactions.py|insert_transactions(...): transaction_number, cyclos_id, date, group_label, from_label, to_label, amount, type_label|MLCFlux stocke une forme analytique minimale : les identifiants transactionnels, la date, les libellés source/cible anonymisés, le montant, et les libellés transactionnels remontés par Cyclos.|« La base locale n’est pas une copie brute de Cyclos ; elle contient le strict nécessaire pour les analyses. »|
|Libellés transactionnels Cyclos conservés|server/sync_transactions.py|group_label, type_label|Ces deux colonnes gardent la trace de la catégorisation transactionnelle issue de Cyclos. MLCFlux peut ensuite les utiliser dans ses règles, tout en construisant ses propres catégories analytiques.|« On conserve les libellés Cyclos d’origine dans la base, mais les familles de graphiques sont ensuite construites par MLCFlux. »|
|État de synchronisation|server/sync_transactions.py / table sync_state|save_sync_state(...): sync_name, last_run_at, last_status, last_message|Chaque grande synchronisation écrit un petit état de suivi : date de dernière exécution, succès ou erreur, message de résultat.|« On garde un état lisible des syncs, ce n’est pas un processus opaque. »|
|Unicité des transactions|base SQLite + server/sync_transactions.py|index unique idx_transactions_cyclos_id_unique ; ON CONFLICT DO UPDATE|Une même transaction Cyclos ne doit pas être dupliquée lors d’un resync. L’identifiant stable cyclos_id sert de garde-fou.|« cyclos_id est la clé de robustesse historique. »|
|Réimport et reclassification sans doublons|server/sync_transactions.py|insert_transactions(...), logique d’UPSERT|Si la logique d’anonymisation ou de classement change, on peut resynchroniser une période : la ligne existante est mise à jour au lieu d’être ajoutée en double.|« Un resync sert aussi à reclasser l’historique proprement. »|
|Exemple concret de reclassification historique|server/utils/anonymizer.py + server/sync_transactions.py|refonte Acteur masqué → T_* et UD_*|Quand on a distingué les comptes techniques et les comptes particuliers de dispositif, l’historique a été rejoué pour mettre à jour les libellés sans changer le nombre final de transactions.|« Les catégories peuvent être corrigées rétroactivement par backfill contrôlé. »|
|Mise à jour planifiée|run_daily_sync.sh + cron|sync script journalier ; sur l’instance dev auditée, cron à 00:00|Les données ne sont pas mises à jour par hasard : un script de synchronisation lance les tâches prévues de façon régulière.|« Les mises à jour passent par un job de sync, pas par l’affichage web. »|
|3. TYPOLOGIE DES ACTEURS ET ANONYMISATION| | | | |
|Catégories internes Cyclos lues sur les acteurs|server/utils/anonymizer.py|lecture de actor.type.internalName dans les objets Cyclos|MLCFlux s’appuie sur le type d’acteur renvoyé par Cyclos pour distinguer notamment compte particulier, compte pro et certains comptes système.|« Pour typer les acteurs, on lit les métadonnées Cyclos de l’acteur, pas seulement son libellé affiché. »|
|Types Cyclos rencontrés dans l’audit|server/utils/anonymizer.py|compteparticulier, comptepro, emission, Conversion|Ces types ont servi à lever l’ambiguïté de l’ancien « Acteur masqué » : certains étaient de vrais comptes techniques, d’autres des comptes particuliers particuliers à un dispositif.|« On a séparé les réalités que Cyclos distinguait déjà implicitement dans ses types internes. »|
|Particuliers ordinaires|server/utils/anonymizer.py|extract_private_actor_key(...), get_or_create_private_pseudo(...), préfixe U_|Les comptes particuliers deviennent des pseudonymes stables du type U_Alice. On ne conserve pas leur identité nominative dans la base analytique.|« Les particuliers sont pseudonymisés dès l’import. »|
|Pourquoi la clé stable repose sur actor.id|server/utils/anonymizer.py|extract_private_actor_key(...): priorité à actor.id ; repli user.id si nécessaire|Dans l’historique, actor.id est plus robuste que user.id pour reconnaître les particuliers présents dans les transactions. Ce choix évite les regroupements artificiels.|« actor.id a été retenu car c’est la clé la plus fiable observée dans l’historique transactionnel. »|
|Mapping persistant des pseudonymes|server/data/user_mapping.json|mapping clé stable → pseudonyme|Le même acteur pseudonymisé garde le même nom affiché d’un import à l’autre, ce qui permet des analyses longitudinales cohérentes.|« L’anonymisation est stable dans le temps, pas recalculée au hasard. »|
|Particuliers de dispositif|server/utils/anonymizer.py ; server/data/device_private_actor_registry.json|is_device_private_actor(...), préfixe UD_|Certains comptes particuliers ont servi à des dispositifs spécifiques. Ils sont séparés en UD_* pour ne pas mélanger un usage événementiel ou temporaire avec les comportements ordinaires.|« UD_* reste un particulier au sens large, mais distinguable analytiquement. »|
|Professionnels|server/utils/anonymizer.py|_extract_professional_code(...), is_professional_label(...), clean_professional_label(...)|Les professionnels sont normalisés autour de leur code Pxxxx. Cela évite que des variations de libellé créent plusieurs acteurs pour un même professionnel.|« On canonise les libellés pro autour du code P. »|
|Comptes techniques|server/utils/anonymizer.py|_technical_actor_label(...), anonymize_actor_label(...)|Les comptes système identifiés dans les transactions sont rendus lisibles sous des libellés analytiques comme T_Émission ou T_Conversion.|« Les comptes techniques ne sont pas fondus avec les usagers ou les pros. »|
|Ordre de reconnaissance des acteurs|server/utils/anonymizer.py|anonymize_actor_label(...): technique / conversion, professionnel, particulier|Le code applique des tests ordonnés pour déterminer le bon type d’acteur. Cela évite des classifications contradictoires.|« Le typage est centralisé et déterministe. »|
|4. VENTILATION ANALYTIQUE DES FLUX| | | | |
|Familles analytiques P / U / T|server/analytics.py|_actor_flow_family(label)|Une fois les libellés stabilisés, MLCFlux les ramène à trois grandes familles simples : professionnels, particuliers, techniques.|« Les familles analytiques servent de langage commun pour tous les indicateurs. »|
|Comptes opérateurs P0000 / P9999|server/analytics.py|_is_operator_account_label(...), _operator_account_code(...)|Ces comptes ont un format P, mais ne représentent pas des professionnels ordinaires. Ils sont isolés pour ne pas fausser l’activité économique de réseau.|« On les conserve, mais on les traite à part. »|
|Fonction centrale de classification|server/analytics.py|_classify_analytical_transaction(row)|C’est le cœur de la ventilation : chaque transaction est orientée vers une catégorie analytique à partir des familles source/cible, des libellés Cyclos et de la présence éventuelle de comptes opérateurs.|« Si vous voulez auditer la logique de classement, c’est cette fonction qu’il faut ouvrir. »|
|Activité économique centrale|server/analytics.py|_classify_analytical_transaction(row), familles U→P, P→P, P→U, U→U|Ce périmètre cherche à mesurer les échanges ordinaires dans le réseau de la Gonette numérique, hors comptes techniques et hors comptes opérateurs.|« C’est notre périmètre de circulation économique centrale. »|
|Alimentations du circuit|server/analytics.py|règles de classement des flux techniques vers usagers/pros|Les entrées de monnaie dans le circuit sont séparées des paiements. Elles ne racontent pas la même chose économiquement.|« Une alimentation n’est pas traitée comme une dépense. »|
|Sorties / reconversions|server/analytics.py|règles de classement des flux de sortie / reconversion|Les flux de retrait ou de reconversion sont suivis dans un bloc propre afin de distinguer création, circulation et sortie du circuit.|« Les sorties sont isolées pour ne pas déformer l’analyse de circulation. »|
|Opérations associatives ou techniques|server/analytics.py|_operations_family_key(...), _operator_operation_profile(...), _operations_functional_flow_key(...)|Ce bloc regroupe les mouvements qui ne sont pas de l’activité économique ordinaire : comptes opérateurs, flux techniques, régularisations ou profils fonctionnels spécifiques.|« On ne jette pas ces flux : on les classe dans un espace analytique adapté. »|
|Agrégats et graphiques|server/analytics.py|compute_global_stats(...), compute_stats_charts(...)|Le backend calcule les indicateurs et les séries graphiques ; le frontend affiche ensuite ces agrégats.|« La doctrine de calcul est côté backend, pas dispersée dans l’interface. »|
|Différence entre catégories Cyclos et catégories MLCFlux|server/sync_transactions.py + server/analytics.py|group_label/type_label conservés ; classifications calculées ensuite|Cyclos fournit des types et libellés transactionnels. MLCFlux les conserve, puis les recoupe avec les familles d’acteurs pour construire des catégories économiques plus lisibles.|« On ne remplace pas les catégories Cyclos : on construit au-dessus une lecture analytique. »|
|5. SOLDES, UTILISATEURS ET DONNÉES CYCLOS COMPLÉMENTAIRES| | | | |
|Soldes utilisateurs récupérables en masse|services / scripts Cyclos de soldes|API /accounts/{accountType}/user-balances ; compteparticulier, comptepro|Pour les soldes, MLCFlux ne passe pas par les transactions une à une : l’API Cyclos expose des soldes par type de compte utilisateur.|« Les soldes utilisent les endpoints user-balances dédiés. »|
|Ce qui remonte dans user-balances|services / scripts Cyclos de soldes|id, balance, group, accountNumber, display ; adresse primaire possible selon option|Ces endpoints fournissent les informations nécessaires pour relier un solde à un utilisateur Cyclos et produire des stocks agrégés.|« Pour les soldes, l’identifiant important est l’id utilisateur renvoyé par Cyclos. »|
|Correspondance actor.id ↔ user.id|server/data/actor_user_links.json et scripts de reconstruction associés|mapping actor_id transactionnel ↔ user_id API utilisateurs / soldes|Les transactions parlent en actor.id ; certaines routes Cyclos utilisateurs/soldes parlent en user.id. MLCFlux construit donc un pont explicite entre les deux.|« actor.id et user.id ne sont pas interchangeables ; le mapping est reconstruit et stocké. »|
|Sync glissante des soldes récents|run_daily_sync.sh + scripts de soldes|fenêtre J-2 → J validée dans les audits|Pour les soldes quotidiens récents, la synchronisation rejoue quelques jours glissants afin de sécuriser les corrections tardives ou retards.|« On resynchronise volontairement une petite fenêtre récente. »|
|Backfill historique des soldes|scripts de backfill des soldes|historique reconstruit depuis 2019-04-17 pour les professionnels ; logique analogue engagée pour les particuliers|Les stocks quotidiens peuvent être reconstruits historiquement par campagnes de backfill, puis entretenus par la synchronisation régulière.|« L’historique n’est pas inféré à l’écran ; il est synchronisé et stocké. »|
|6. DONNÉES ODOO COMPLÉMENTAIRES| | | | |
|Enrichissement des professionnels via Odoo|server/sync_odoo_professional_enrichment.py|extraction des Pxxxx → récupération d’informations Odoo → remplacement du snapshot local|MLCFlux peut enrichir la fiche des professionnels avec des informations issues d’Odoo, en gardant cette couche distincte des transactions Cyclos.|« Cyclos fournit les flux ; Odoo peut enrichir les métadonnées pro. »|
|Masse monétaire et garanties issues d’Odoo|server/services/odoo_monetary_indicators.py ; server/sync_odoo_monetary_indicators.py|account.move.line/read_group ; stock cumulé à fin d’année date <= Y-12-31|Les indicateurs de masse monétaire et de fonds de garantie ne viennent pas de Cyclos mais de la comptabilité Odoo, avec une logique de stock cumulé à date.|« Les flux numériques et les grandeurs comptables ne viennent pas de la même source. »|
|Séparation méthodologique Odoo / Cyclos|architecture MLCFlux|transactions/soldes Cyclos ≠ comptabilité Odoo|MLCFlux évite de mélanger les sources : Cyclos sert à analyser les usages numériques ; Odoo sert aux grandeurs comptables ou aux enrichissements métier pertinents.|« Les sources sont articulées, mais jamais confondues. »|
|7. QUESTIONS DE ROBUSTESSE PROBABLES| | | | |
|Que se passe-t-il si les catégories Cyclos évoluent ?|server/utils/anonymizer.py + server/analytics.py|règles centralisées de typage et de classification|Si un type ou un libellé Cyclos change, le point de correction est localisé : on ajuste la règle centrale, puis on peut resynchroniser l’historique.|« Le modèle est maintenable parce que les règles ne sont pas disséminées. »|
|Peut-on auditer la logique de calcul ?|server/analytics.py|_classify_analytical_transaction(...), compute_global_stats(...), compute_stats_charts(...)|Oui : la classification et les agrégations sont dans des fonctions identifiables, et non cachées dans le frontend.|« Le cœur méthodologique est auditable côté backend. »|
|Peut-on répondre sans être développeur expert ?|positionnement de présentation|—|Tu peux assumer que la conception a été réalisée avec l’aide de LLM, tout en montrant que les sources, les règles et les contrôles ont été pensés, testés et stabilisés.|« Je ne prétends pas avoir codé seul chaque ligne ; je peux expliquer la logique de données, les choix méthodologiques et les points de contrôle. »|
